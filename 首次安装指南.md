# 首次安装指南（从零搭建）

本指南面向**第一次在一台新电脑上搭建本框架**的人。按顺序做完即可跑起来。

> 适用系统：**macOS**。框架内置了 macOS 相关假设（微信 TCC 容器探测、`/Volumes` NAS 挂载、
> Chrome 远程调试）。Windows / Linux 需自行适配，本指南不覆盖。

---

## 0. 环境要求一览

| 组件 | 版本 / 说明 | 用途 |
|---|---|---|
| **Python** | 3.11（两层都要求 `>=3.11`） | 运行时 |
| **uv** | 最新版 | 包管理 / 建虚拟环境（比 pip 快，项目统一用它） |
| **Google Chrome** | 任意稳定版 | 平台登录态学习（9222 远程调试 + Playwright CDP） |
| **Playwright 浏览器** | chromium（由 Ops-Cli 安装） | 平台页面自动化 |
| Hermes 通知 agent | 可选 | 微信 / 飞书 / 企业微信通知；不装则通知功能不可用，不影响主流程 |
| MCP code-review-graph | 可选（仅开发用） | 代码图谱，加速 AI 代码导航；与运行无关 |

本框架坚持**两层各自独立 venv**（不要混用）：

- 平台层 `运营自动化项目/Ops-Cli/.venv`
- 业务层 `运营自动化项目/运营自动化工具/.venv`

---

## 1. 安装 uv

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
# 装完重开终端，确认：
uv --version
```

（已装可跳过。也可用 Homebrew：`brew install uv`。）

---

## 2. 业务层：建 venv 并装依赖

```bash
cd 运营自动化项目/运营自动化工具
uv venv --python 3.11 .venv
uv pip install -r requirements.txt pytest
```

业务层依赖很轻：`openpyxl`（Excel）、`Pillow`（图片）、`requests`。

> 业务层有「venv 自举」：`run.py` 顶部会在执行任何业务逻辑前 re-exec 到本层 `.venv`，
> 所以手动跑和定时跑都收敛到同一解释器。文档里照常写 `python3 run.py` 即可。

---

## 3. 平台层：建 venv、装依赖、装浏览器

```bash
cd 运营自动化项目/Ops-Cli
uv venv --python 3.11 .venv
uv pip install -e ".[dev]"          # 装 ops 命令本体 + 依赖（含 playwright、typer、PyYAML…）
.venv/bin/playwright install chromium   # 下载 Playwright 用的 chromium
```

装完确认 `ops` 命令可用：

```bash
.venv/bin/ops --help
```

> 业务层通过 `clients/ops_cli_client.py` 唯一桥接调用 `ops --json`；
> 默认会找 `Ops-Cli/.venv/bin/ops`，所以这一步的 venv 路径不要乱挪。

---

## 4. 路径配置（零配置 / 一键体检）

框架的路径全部从**锚点推导**，换电脑 / 换用户名默认零配置即可跑。首次建议跑一次体检：

```bash
cd 运营自动化项目/运营自动化工具
python3 run.py setup
```

它会：体检全部路径 → 探测微信 / NAS → 生成 `config/paths.local.yaml`（本机覆盖，已 gitignore）。

- 唯一可能要手填的是 **NAS 挂载点**，且仅 NAS 相关 workflow 需要。
- 模板见 `config/paths.yaml.example`；**绝不要提交** `paths.local.yaml` / `paths.yaml`。

---

## 5. 密钥配置（按需）

凡 `*.example` 模板，复制一份去掉 `.example` 再填值，**真实密钥文件一律 gitignore，绝不提交**：

```bash
cd 运营自动化项目/运营自动化工具/config
cp ai_iterate.secret.env.example ai_iterate.secret.env   # 需要 AI 迭代功能时才填
```

红线：任何 Cookie / Token / Authorization / 账号密码，都不写进 `.md` / `.yaml` / `.json` 文档。

---

## 6. 平台登录态（关键一步：scene 学习）

平台数据要靠真实浏览器登录态。**框架不会、也不能附带任何人的 Cookie**——每台机器要自己抓。

```bash
# 1) 用框架脚本起一个带远程调试(9222)的 Chrome
bash 运营自动化项目/Ops-Cli/sessionhub/browser/start_chrome_9222.sh

# 2) 在这个 Chrome 里手动登录目标平台（如猫超后台 / 聚水潭）

# 3) 抓取对应场景（示例，按实际平台/命令）：
cd 运营自动化项目/Ops-Cli
.venv/bin/ops tmcs product learn      # 学习"商品列表"场景
.venv/bin/ops jst auth check          # 校验聚水潭登录态
```

- 站点 / 场景定义在 `Ops-Cli/sessionhub/config/sites/<平台>.yaml`，是唯一来源。
- 接**新平台**：加 `Ops-Cli/src/ops_cli/platforms/<x>/platform.py`（实现 `register()`）
  + `config/sites/<x>.yaml`，核心调度自动发现，无需改框架。

## 7. 验证安装

```bash
# 业务层测试（应全过）
cd 运营自动化项目/运营自动化工具 && .venv/bin/python -m pytest -q

# 平台层测试
cd 运营自动化项目/Ops-Cli && .venv/bin/python -m pytest -q

# 跑一个 workflow 的 dry-run（不触发任何真实平台动作 / 通知）
cd 运营自动化项目/运营自动化工具
python3 run.py --list                      # 看所有任务
python3 run.py workflow <某 workflow_id> --dry-run
```

dry-run 永远安全：不下载平台文件、不写主数据、不发通知、不动 NAS。

## 8. 可选组件

### 通知（Hermes）
通知统一走 `core.runtime.send_notification`，后端是**可插拔注册表**。默认参考后端
（企业微信 wecom / Hermes）在 `core/runtime/notify_backends.py`。不装 Hermes 时，
通知功能不可用，但所有 dry-run 和非通知流程照常工作；要换钉钉 / 邮件，
`register_sender("名字", 工厂)` 即可，不改引擎。

### AI 知识库
见同级 `../ai知识库/`。结构与机制已就位，正文用业务层 workflow 生成：

```bash
cd 运营自动化项目/运营自动化工具
python3 run.py workflow ai_knowledge_base_update \
  --source-root <运营自动化项目绝对路径> \
  --kb-root <ai知识库绝对路径>
```

### 开发用代码图谱（MCP code-review-graph）
仅在用 Claude Code / Codex 开发本框架时有用，与运行无关，可不装。

## 常见问题

| 现象 | 处理 |
|---|---|
| `ops: command not found` | 确认在 `Ops-Cli/.venv/bin/ops` 下调用，或 `uv pip install -e .` 没装成功 |
| 业务依赖缺失（如 `No module named openpyxl`） | 没装进**业务层** venv；别借用 Ops-Cli 的 venv，重做第 2 步 |
| 平台命令报登录态失效 | 重做第 6 步：起 9222 Chrome → 手动登录 → 重新 learn |
| 路径找不到 | 跑 `python3 run.py setup`，或在 `config/paths.local.yaml` 手填 |
| 微信目录弹隐私授权框 | 正常（macOS TCC）；只有用到微信的 workflow 才会触发，按需授权 |
